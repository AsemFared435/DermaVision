Graduation Project

DermaVision

AI-assisted dermatology platform for preliminary skin image analysis, safe medical explanation, and consultation reports.

Project Overview

DermaVision allows users to upload a skin image, receive an AI-based preliminary diagnosis, ask diagnosis-aware questions through a safe RAG assistant, and generate printable reports for educational and consultation purposes.

Backend Technology Choice

Why We Chose FastAPI

API-first and ML-friendly

The project is mainly a REST API around image upload, ML inference, auth, history, and RAG. FastAPI fits this style naturally.

Automatic API Docs

Swagger UI at /docs helps test endpoints during development and viva discussion.

Pydantic Validation

Request and response schemas protect the API from invalid data and make contracts clear for the frontend.

Async Support

Useful for database calls, file operations, email sending, and RAG workflows without blocking the server.

Why not Flask?

Flask is simple, but FastAPI gives typing, validation, dependency injection, and docs with less custom work.

Why not Django?

Django is powerful but heavier for an API-focused ML backend. FastAPI keeps the backend lighter and more modular.

Backend Patterns

Design Patterns Used

Layered Architecture

Separates endpoints, services, infrastructure, and database code.

Service Layer

Business workflows live in services such as DiagnosisService and RagService.

Repository Pattern

Repositories hide database details and provide clean data-access methods.

DTO / Mapper Pattern

Schemas and mappers keep response shapes consistent for the frontend.

Provider / Strategy

RAG can switch between Groq, xAI, OpenAI, and local fallback using configuration.

Fallback Pattern

If an LLM fails, the deterministic local generator still returns a safe response.

Dependency Injection

FastAPI dependencies provide current user, DB session, and service instances.

Middleware Pattern

Cross-cutting concerns like logging, rate limits, and security headers are centralized.

Centralized Config

Secrets, URLs, model path, CORS, limits, and providers are controlled by environment variables.

AI Diagnosis

EfficientNet-B4 Prediction Pipeline

Why EfficientNet-B4?

  • Good balance between accuracy and model size.
  • Strong image feature extraction for medical image classification.
  • Works well with transfer learning and PyTorch/timm workflows.

What the Model Returns

  • Predicted disease label.
  • Confidence score.
  • Top predictions.
  • Image quality score and quality label.
  • Analysis ID for history, result refresh, RAG chat, and reports.

Supported Classes

Urticaria Tinea Circinata Annular Lichen / Lichen Planus Psoriasis Mycosis Fungoides / MF Healthy

Important Concept

Model predicts, RAG explains. The ML model classifies the image. The RAG assistant explains the condition using a controlled knowledge base and safety rules.

Database

Database Design

The project uses one application database locally: SQLite. SQLAlchemy async ORM and Alembic migrations keep the schema organized and portable.

TableMain ColumnsPurpose
usersid, email, username, hashed_password, is_active, created_at, updated_atStores user accounts and authentication identity.
diagnosesid, user_id, family_member_id, session_id, image_path, image_quality_score, image_quality_label, disease_type, probability, confidence_percentage, all_predictions, extra_metadata, created_at, updated_atStores each diagnosis result and links it to the owner.
family_membersid, user_id, name, relation, date_of_birth, gender, notes, created_at, updated_atAllows users to run analysis for themselves or family members.
password_reset_tokensid, user_id, token_hash, expires_at, used_at, created_atStores hashed single-use password reset tokens.
alembic_versionversion_numInternal migration tracking table, not business data.

Why SQLite?

Simple local/demo setup with no external database server.

Why Alembic?

Tracks schema changes through migrations instead of manual SQL edits.

Production Note

PostgreSQL is recommended for production scalability and reliability.

RAG Backend

RAG Architecture and What We Added

1Diagnosis ContextLoad diagnosis by analysis_id and verify ownership.
2Intent DetectionDetect diagnosis, treatment, urgent care, contagion, dosage, age group, and compound questions.
3Knowledge LoadingLoad skin_data1.txt as the dermatology knowledge base.
4ChunkingSplit the knowledge file into searchable chunks.
5Local IndexBuild a lightweight TF-IDF-like vector index with disease tags.
6RerankingUse predicted_label as a strong signal so sources match the selected disease.
7GenerationUse Groq/xAI/OpenAI if configured, otherwise local deterministic fallback.
8ValidationPost-generation validator rejects unsafe or inaccurate medical answers.

RAG Safety

Medical Guardrails in the Assistant

Deterministic Safety Profiles

  • Canonical disease names in English and Arabic.
  • Urticaria is forced as not contagious.
  • Tinea is forced as contagious.
  • Urgent red flags are controlled by backend profiles.

Provider Safety

  • Groq, xAI, and OpenAI are optional.
  • No API keys are exposed in the frontend.
  • Out-of-scope questions do not call LLM generation.
  • If provider fails, local fallback still answers safely.

No Personal Prescription

  • No exact dose.
  • No frequency.
  • No treatment duration.
  • Strong treatments are only under medical supervision.

Post-generation Validation

  • Blocks wrong translations like Urticaria as nausea.
  • Rejects false contagion claims for Urticaria.
  • Removes unsafe wording and falls back to deterministic answers.

Reports

Report System

Preliminary Report

  • Generated from the diagnosis result page.
  • Based on the AI diagnosis result and general educational guidance.
  • Downloadable or printable from the result page.
  • Does not call the RAG consultation report endpoint.

Consultation Report

  • Generated after the chatbot conversation.
  • Uses diagnosis context and summarized patient concerns.
  • Includes general care guidance, red flags, and disclaimer.
  • Can be printed or copied from the chat report modal.

Medical wording rule

We use "Preliminary Report" and "Consultation Report". We avoid wording like final diagnosis, recommended treatment, or personal prescription.

Security and Privacy

Security Features

Authentication

JWT auth protects private endpoints and redirects users when a token expires.

Ownership Checks

Users can only access their own diagnoses, RAG chat context, and generated reports.

Password Security

Passwords are hashed. Reset tokens are hashed, expiring, and single-use.

Upload Security

Only JPG, JPEG, PNG are accepted. Max size is 10MB. Real image content is verified.

Rate Limiting

Sensitive routes such as login, signup, reset, diagnosis, and RAG chat are rate limited.

Secrets Hygiene

LLM and SMTP keys are backend-only. .env is ignored by Git. Examples use placeholders only.

DevOps

Docker and GitHub Readiness

Docker Setup

  • Backend Dockerfile runs FastAPI on port 8000.
  • Frontend Dockerfile builds Vite and serves with nginx.
  • docker-compose.yml runs backend and frontend together.
  • Persistent folders can be mounted for SQLite, uploads, and RAG index cache.

GitHub Safety

  • .gitignore and .dockerignore exclude secrets and generated files.
  • No real API keys or SMTP passwords are committed.
  • README files describe local and Docker setup.
  • Model weights are around 67.7 MiB, so Git LFS is recommended for a polished public repository.

Deployment note

For production, use a strong SECRET_KEY, AUTH_REQUIRED=true, strict CORS origins, backend-only LLM keys, and PostgreSQL instead of SQLite.

Medical Responsibility

Medical Safety and Limitations

What the System Can Do

  • Provide a preliminary AI-assisted prediction.
  • Explain supported conditions using a local knowledge base.
  • Highlight red flags and when to seek urgent care.
  • Generate educational reports for consultation preparation.

What the System Cannot Do

  • It cannot replace a dermatologist.
  • It cannot provide a final medical diagnosis.
  • It cannot prescribe personal medication.
  • It cannot decide exact dose, frequency, or duration.

Core disclaimer

DermaVision is an educational and assistive system. A dermatologist must confirm the diagnosis and decide the correct treatment.
