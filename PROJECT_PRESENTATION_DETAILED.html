Graduation Project Detailed Notes

DermaVision

منصة ويب ذكية مساعدة في مجال الجلدية تجمع بين تشخيص مبدئي من صورة جلدية، ومساعد RAG آمن، وتقارير قابلة للطباعة، مع بنية Backend/Frontend جاهزة للعرض والنشر.

AI Diagnosis EfficientNet-B4 predicts the skin condition and confidence.
Safe RAG Assistant Arabic/English chatbot grounded in dermatology knowledge.
Secure Full Stack JWT auth, SQLite/Alembic, Docker, reports, and history.

1. الفلو العام للمشروع

الفكرة الأساسية أن الموديل يتوقع المرض من الصورة، ثم RAG يشرح النتيجة للمستخدم بطريقة تعليمية وآمنة.

1تسجيل أو دخول
المستخدم يعمل حساب أو يسجل دخول، ويحصل على JWT Token.
2رفع صورة الجلد
الواجهة ترسل الصورة للـ Backend مع التوكن.
3تحقق من الصورة
الـ Backend يتحقق من النوع والحجم وأن الملف صورة حقيقية.
4تشخيص مبدئي
EfficientNet-B4 يرجع predicted label وconfidence وtop predictions.
5حفظ النتيجة
النظام يحفظ التشخيص والصورة والـ metadata في قاعدة البيانات.
6عرض التقرير
المستخدم يرى النتيجة، جودة الصورة، الاحتمالات، ومعلومات عامة.
7اختيار الخطوة التالية
إما تحميل Preliminary Report أو التحدث مع المساعد.
8RAG Chat + Consultation Report
المساعد يجيب من قاعدة المعرفة، وبعد الحوار يمكن إنشاء Consultation Report.
مهم في المناقشة: النظام لا يدعي تشخيصًا نهائيًا. كل النتائج مبدئية وتعليمية، ويجب تأكيدها مع طبيب جلدية.

2. Project Overview

  • DermaVision منصة Full-Stack لتحليل صور الجلد بالذكاء الاصطناعي.
  • المستخدم يرفع صورة جلدية ويحصل على نتيجة مبدئية مع نسبة ثقة وجودة الصورة.
  • النظام يوفر مساعد RAG يشرح الحالة، العدوى، الخطورة، العناية العامة، والأسئلة المناسبة للطبيب.
  • يوجد نظام تقارير: Preliminary Report من صفحة النتيجة، وConsultation Report بعد المحادثة.
  • كل المحتوى الطبي تعليمي ومساعد فقط، وليس بديلًا عن الطبيب.

3. لماذا اخترنا FastAPI في الـ Backend؟

مناسب للـ AI APIs FastAPI ممتاز في بناء APIs سريعة حول ML inference، خصوصًا مع upload files وasync endpoints.
Swagger تلقائي يعطي Documentation تلقائي على /docs لتجربة endpoints في المناقشة.
Pydantic Validation يساعد في request/response validation وتقليل أخطاء البيانات.
Async Support مفيد مع database async، file handling، وعمليات RAG أو email.
أخف من Django Django قوي جدًا لكنه أكبر من المطلوب هنا، لأن المشروع API-focused وليس CMS أو Admin-heavy.
أكثر تنظيمًا من Flask للمشروع ده Flask بسيط، لكن FastAPI أعطانا typing, validation, docs, dependency injection بشكل جاهز.

4. Architecture / Structure

الـ Backend معمول بأسلوب Layered / Clean Architecture بحيث كل طبقة لها مسؤولية واضحة. هذا يجعل المشروع أسهل في الشرح، الاختبار، والتطوير.

FrontendReact/Vite UI
API LayerFastAPI endpoints
Service LayerBusiness logic
InfrastructureML, RAG, Email
DatabaseSQLAlchemy/Alembic

Backend Layers

  • API Layer: ملفات app/api/v1/endpoints تستقبل الطلبات وترجع responses.
  • Domain/Service Layer: ملفات مثل diagnosis_service.py وrag_service.py تنفذ الفلو والمنطق.
  • Infrastructure Layer: ML model، RAG modules، email service، database session، providers.
  • Database Layer: SQLAlchemy models وrepositories مثل DiagnosisRepository.

Frontend Structure

  • Pages: Home, Upload, Result, Chat, Dashboard, History, Auth pages.
  • Components: Navbar, Footer, Hero, disease cards, family manager, etc.
  • Services: authService, diagnosisService, chatService, familyService.
  • Contexts: Auth, Language, Theme.
  • Localization: Arabic/English text and RTL support.

5. Design Patterns Used

Layered / Clean Architecture فصلنا API عن business logic عن infrastructure، عشان أي جزء يتغير بدون كسر باقي النظام.
Service Layer Pattern مثال: DiagnosisService يدير رفع الصورة، prediction، والحفظ. RagService يدير chat/report flow.
Repository Pattern مثال: DiagnosisRepository يخفي تفاصيل SQLAlchemy ويقدم functions مثل create/get_by_user.
DTO / Mapper Pattern mappers تحول database objects إلى API response shape ثابت ومناسب للـ Frontend.
Provider / Strategy Pattern RAG يمكنه التبديل بين groq أو xai أو openai أو local fallback بدون تغيير endpoint.
Fallback / Resilience Pattern لو LLM provider فشل أو API key غير موجود، النظام يرجع لإجابات deterministic آمنة بدل ما ينهار.
Centralized Configuration كل الإعدادات مثل DB URL وCORS وLLM providers وSMTP تأتي من environment variables.
Middleware Pattern Security headers، logging، وrate limiting يتم تطبيقهم على الطلبات مركزيًا.

6. AI Diagnosis

  • الموديل المستخدم: EfficientNet-B4.
  • الـ inference يتم في Backend باستخدام PyTorch/timm.
  • الموديل لا يشرح المرض؛ دوره فقط prediction من الصورة.
  • RAG هو المسؤول عن شرح النتيجة للمستخدم.

Supported Classes

Urticaria Tinea Circinata Annular Lichen / Lichen Planus Psoriasis Mycosis Fungoides / MF Healthy

Diagnosis Output

  • predicted_label: اسم الحالة المتوقعة.
  • confidence: نسبة ثقة الموديل.
  • top_k / all_predictions: أعلى احتمالات.
  • image_quality_score: تقييم جودة الصورة من 0 إلى 100.
  • analysis_id: رقم التحليل المستخدم في history وRAG chat وreports.

8. لماذا SQLite محليًا؟ ولماذا Alembic؟

SQLite مناسب للعرض والتطوير لا يحتاج Database server منفصل، وهذا يسهل تشغيل المشروع في viva أو Docker demo.
SQLAlchemy يحافظ على قابلية النقل لأننا نستخدم ORM، يمكن الانتقال إلى PostgreSQL في الإنتاج بدون إعادة كتابة كل logic.
Alembic لإدارة schema يحفظ تطور الجداول عبر migrations بدل إنشاء الجداول يدويًا.
PostgreSQL أفضل للإنتاج لأن الإنتاج يحتاج concurrency أعلى، backups، مراقبة، وأداء أقوى من SQLite.

9. RAG Architecture: ماذا أضفنا ولماذا؟

RAG في DermaVision ليس Chat عام. هو Diagnosis-aware assistant: يبدأ من analysis_id، يتأكد من ملكية المستخدم للتشخيص، ثم يستخدم predicted disease لتوجيه البحث والإجابة.

RAG Flow

1Request
POST /api/v1/rag/chat يستقبل analysis_id وmessage وlanguage.
2Ownership Check
RagService يحمل diagnosis ويتأكد أن diagnosis.user_id هو نفس المستخدم.
3Intent Detection
local keyword detector + optional semantic classifier لتحديد هل السؤال diagnosis, treatment, contagion, urgent, etc.
4Disease-aware Retrieval
retriever يعطي أولوية للمقاطع المرتبطة بالمرض المتوقع، مثل Urticaria أو Tinea.
5Safe Generation
إما deterministic local generator أو Groq/xAI/OpenAI provider حسب الإعدادات.
6Validation
post-generation validation يرفض الإجابات الخطرة أو الترجمات الطبية الخاطئة.

RAG Modules

ModuleRole
document_loader.pyتحميل ملف المعرفة skin_data1.txt.
text_splitter.pyتقسيم النص إلى chunks مناسبة للبحث.
vector_store.pyفهرس محلي lightweight يشبه TF-IDF ويخزن disease tags.
retriever.pyاسترجاع chunks مرتبطة بالمرض والنية، مع reranking.
intent.pyكشف intents محليًا: diagnosis, treatment, contagion, urgent, dosage, age groups, etc.
semantic_intent_classifier.pyتصنيف intent بالـ LLM اختياريًا لو provider متاح.
generator.pyFallback deterministic generator لإجابات آمنة بدون API keys.
disease_profiles.pycanonical names وحقائق safety حتمية لكل مرض.
groq_generator.py, xai_generator.py, openai_generator.pyProvider strategy لاستخدام LLM اختياريًا.
rag_service.pyينسق كل الفلو: diagnosis context, retrieval, intent, generation, validation, final report.

لماذا أضفنا RAG؟

  • الموديل يتوقع label فقط، ولا يشرح المرض للمستخدم.
  • RAG يجعل الشرح grounded في ملف معرفة محلي بدل إجابة عامة من LLM.
  • RAG يقلل hallucination لأن الإجابة مرتبطة بمصادر وسياق تشخيص المستخدم.
  • أضفنا deterministic safety profiles لأن بعض الحقائق الطبية لا يجب تركها للـ LLM.

10. Medical Safety Guardrails

  • لا توجد وصفة شخصية، ولا جرعات، ولا عدد مرات استخدام، ولا مدة علاج شخصية.
  • أسئلة الخطورة مثل "ممكن أموت؟" تتحول إلى urgent-care intent.
  • Urticaria يتم إجبارها كحالة غير معدية.
  • Tinea يتم إجبارها كحالة معدية لأنها عدوى فطرية.
  • canonical disease names تمنع أخطاء ترجمة مثل ترجمة Urticaria بشكل خاطئ.
  • post-generation validation يرفض عبارات خطرة أو غير دقيقة من LLM.
  • المستخدم يرى disclaimer أن النظام للتوعية ولا يغني عن طبيب جلدية.
الفكرة المهمة: LLM لا يأخذ القرار الطبي وحده. الحقائق عالية الخطورة تأتي من disease_profiles deterministic rules.

11. Security Features

JWT Authenticationالتوكن يرسل مع الطلبات المحمية مثل diagnosis/history/RAG.
Protected Endpointsالـ API يعتمد على get_current_user_id لحماية البيانات.
Ownership Checksالمستخدم لا يستطيع فتح diagnosis أو report لا يخصه.
Password Hashingكلمات المرور تحفظ hashed فقط.
Password Reset Tokensreset token يتم تخزينه hashed، single-use، وله expiry.
Upload SecurityJPG/JPEG/PNG فقط، 10MB limit، MIME validation، وPillow image verification.
Rate Limitingمطبق على login/signup/forgot/reset/diagnosis/RAG chat.
Security Headersmiddleware يضيف headers تقلل مخاطر browser-side attacks.
CORSالسماح فقط بالـ frontend origins المطلوبة.
Secrets HygieneAPI keys وSMTP/JWT secrets في backend env فقط، و.env خارج Git.

12. Report System

Preliminary Report

  • ينشأ من صفحة نتيجة التشخيص.
  • يعتمد على AI diagnosis والـ basic report الموجود في الصفحة.
  • قابل للطباعة أو الحفظ PDF.
  • لا يسمى final diagnosis ولا personal prescription.

Consultation Report

  • ينشأ من صفحة chat بعد المحادثة.
  • يرسل chat messages + analysis_id إلى POST /api/v1/rag/final-report.
  • يتضمن diagnosis context، summary لأسئلة المريض، general care guidance، red flags، وdisclaimer.
  • لا يظهر كتشخيص نهائي؛ هو تقرير توعوي للاستشارة.

16. Short Presentation Summary

DermaVision is a full-stack AI-assisted dermatology system. It combines image-based skin disease prediction, secure user management, diagnosis history, a safe RAG medical assistant, and printable reports. The backend follows layered clean architecture with service, repository, DTO, middleware, and provider strategy patterns. The system supports Arabic and English, Docker-based setup, optional Groq/xAI/OpenAI generation, and deterministic medical safety guardrails so the output remains educational and not a personal prescription.

Final viva sentence: "Model predicts, RAG explains, and safety guardrails control the medical wording."
